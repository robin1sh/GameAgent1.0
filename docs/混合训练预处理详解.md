# 混合训练预处理详解（Mario + CoinRun）

本文档面向当前项目的 `unified mixed` 训练实现，系统梳理预处理链路及其设计原因，便于实验复现、论文撰写与参数排查。

---

## 1. 预处理的核心目标

混合训练中的预处理主要解决四个问题：

1. 输入统一：让 Mario 与 CoinRun 都能喂给同一个 CNN。
2. 时序可见：单帧图像不含速度信息，必须补充短时动态线索。
3. 训练稳定：多任务奖励量级差异会导致梯度偏向，需要做奖励尺度治理。
4. 复现可靠：训练与评估包装必须一致，否则指标会失真。

---

## 2. 观测预处理：从原始帧到网络输入

## 2.1 Mario 分支（`envs/mario_env.py`）

处理链路：

1. `JoypadSpace(COMPLEX_MOVEMENT)`
2. `SkipFrameWrapper(skip=4)`（动作重复执行 + 奖励累积）
3. `GrayScaleObservation(keep_dim=True)`（RGB -> 灰度，保留通道维）
4. `ResizeObservation((84, 84))`（统一分辨率）
5. `TimeLimit(max_episode_steps)`（统一回合长度上限）
6. `ActionMappingWrapper("mario")`（映射到统一动作语义）

设计原因：

- 灰度化：减少颜色冗余，降低输入维度和计算压力。
- 84x84 缩放：兼顾空间信息与训练效率，是视觉 RL 常见折中。
- skip=4：降低决策频率，减少相邻帧冗余，提升吞吐。
- `keep_dim=True`：保证观测仍是三维图像张量，便于后续堆叠与 CNN 输入。

## 2.2 CoinRun 分支（`envs/coinrun_env.py` + `envs/jumper_env.py`）

处理链路（核心在 `ProcgenObsWrapper`）：

1. 从 Procgen 输出提取 `rgb`
2. 灰度化 + 缩放到 `84x84`
3. 与 Mario 对齐的 skip-frame（默认 4），窗口内奖励累积
4. 将彩色帧写入 `info["rgb"]`（用于可视化或采集，不参与训练输入）

设计原因：

- Procgen 默认输出格式与 Mario 不一致，必须统一到同一观测语义。
- 采用同样的 `84x84 + skip=4`，让两任务共享同一视觉编码器时分布差更可控。
- 保留 `info["rgb"]` 是工程上的可观测性设计，便于调试，不污染训练输入。

---

## 3. 帧堆叠与张量布局

在向量环境层（`make_vec_env` / `make_mixed_vec_env`）两边都会执行：

- `VecFrameStack(n_stack=4, channels_order="last")`

得到单环境观测形状：`(84, 84, 4)`（4 帧沿通道维拼接）。

训练脚本再做：

- `VecTransposeImage`：把 `HWC` 转成 `CHW`，以适配 PyTorch 卷积输入。

设计原因：

单帧很难推断速度和方向变化。堆叠 4 帧后，网络可从像素变化中恢复短时运动信息，决策更稳定。

---

## 4. 混合环境拼接前后的“一致性检查”（`envs/mixed_env.py`）

`MixedVecEnv` 会做两层检查：

1. 空间级检查（初始化时）
   - `observation_space.shape` 必须一致
   - `observation_space.dtype` 必须一致
   - 动作空间必须一致
2. 运行时检查（首次 `reset`）
   - 实际 `obs` 维度、shape、dtype 一致
   - 无 NaN/Inf
   - 值域合法（`uint` 常见在 `[0, 255]`）

设计原因：

多任务训练里最隐蔽的问题是“能跑但错位”（例如 HWC/CHW 混淆、dtype 不一致、观测脏值）。  
这些问题若不提前拦截，通常表现为“训练不收敛但也不报错”，排查成本很高。

---

## 5. 动作预处理：统一动作语义（`envs/action_mapping.py`）

策略网络输出统一离散动作（15 维语义空间）：

- Mario：通过映射表转换到 `COMPLEX_MOVEMENT`（12 动作）
- CoinRun/Procgen：保持 `0~14` 动作语义

设计原因：

混合训练的本质是“一个策略头服务多个任务”。  
动作语义不统一时，策略输出层无法共享，训练目标会冲突。

---

## 6. 奖励预处理：混合训练稳定性的关键

## 6.1 CoinRun 奖励塑形（`MixedVecEnv` 默认路径）

在 `use_aligned_reward=False` 时，CoinRun 使用塑形奖励：

- `raw_reward * coinrun_reward_scale`
- `+ max(0, x_now - x_prev) * coinrun_progress_coef`（仅奖励向右进度）
- `- coinrun_step_penalty`
- 通关 `+ coinrun_success_bonus`
- 失败终止 `- coinrun_fail_penalty`

Mario 分支通常保留原始奖励语义（含 skip-frame 累积效果）。

设计原因：

- Mario 与 CoinRun 原始奖励分布不同，直接混训会让某一任务主导梯度。
- “只奖励向右增量”可抑制左右抖动刷分。
- 时间惩罚鼓励策略减少无效拖延。

## 6.2 训练期奖励归一化（`train_model/train_unified.py`）

默认启用：

- `VecNormalize(norm_obs=False, norm_reward=True, clip_reward=10.0, gamma=0.95, training=True)`

并在训练结束保存：

- `best_model/<exp-id>/vecnormalize.pkl`

设计原因：

当前观测已经是受控的灰度图像分布（范围明确、预处理一致）。  
主要不稳定来源在多任务奖励尺度，因此优先做 `norm_reward=True`。

---

## 7. 训练与评估的一致性要求（非常重要）

测试脚本（`test/test_unified_PPO_model.py`）与训练保持同样包装：

1. `make_vec_env(...)`
2. `VecMonitor + VecTransposeImage`
3. 若训练启用了奖励归一化，则评估时必须加载 `vecnormalize.pkl`，并设置：
   - `vec_env.training = False`
   - `vec_env.norm_reward = False`

设计原因：

如果评估阶段不加载同一份归一化统计，输入/奖励统计口径会变化，指标不可比，结论会失真。

---

## 8. 一句话数据流总览

`原始游戏帧 -> 灰度/缩放 -> 跳帧累积 -> 4帧堆叠 -> 混合拼接 -> HWC->CHW -> CustomCNN -> PPO`

---

## 9. 常见问题与排查建议

1. 报 `shape/dtype` 不一致：先看 `MixedVecEnv` 一致性报错，通常是某一分支 wrapper 顺序变化。
2. 训练抖动大：先检查 `coinrun_reward_scale` 与 `--normalize-reward` 是否匹配。
3. 评估结果和训练日志对不上：检查是否加载对应 `vecnormalize.pkl`。
4. 回报上涨但通关不上升：回报提升不等于成功率提升，需单独看通关指标。

---

## 10. 复现检查清单（Checklist）

- [ ] Mario 与 CoinRun 都使用 `84x84` 灰度观测
- [ ] 两分支都使用 `skip_frames=4`
- [ ] 两分支都使用 `VecFrameStack(4)`
- [ ] 混合训练中观测/动作空间一致性检查通过
- [ ] 奖励塑形参数已记录（尤其 CoinRun）
- [ ] 是否启用 `--normalize-reward` 已记录
- [ ] 若启用归一化，`vecnormalize.pkl` 已保存并在评估时加载
- [ ] 训练与评估包装链路一致

---

## 11. 结论

本项目的混合训练预处理可概括为：

**统一输入表征（灰度 + 缩放 + 堆叠） + 统一动作语义 + 奖励尺度治理（塑形 + 归一化） + 一致性校验与评估同口径。**

这四部分共同保证了 Mario/CoinRun 在共享策略下可训练、可解释、可复现。
