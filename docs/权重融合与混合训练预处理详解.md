# 权重融合与混合训练预处理详解（Mario + CoinRun）

> 目的：把“从单任务 PPO 到权重融合，再到 mixed 续训”这条主线中所有关键预处理一次讲清楚，并说明每一步为什么要做、如果不做会有什么风险。

---

```text
  -> Unified 模型续训与对比评估
```

## 3. 预处理总表（做法 + 原因 + 风险）

| 预处理项 | 当前做法 | 为什么要做 | 不做会怎样 |
|---|---|---|---|
| 观测灰度化 | RGB 转 1 通道灰度 | 降低冗余颜色信息，提升样本效率 | 输入维度更大，训练更慢更不稳 |
| 观测缩放 | 统一到 `84x84` | 统一 CNN 输入口径，便于跨任务共享结构 | 两任务输入尺度不一致，难以复用 |
| 跳帧 | `skip=4` 动作重复+奖励累积 | 降低决策频率，减少计算开销 | 训练成本上升，噪声更大 |
| 帧堆叠 | `VecFrameStack(n_stack=4)` | 给策略提供短时动态信息（速度/方向） | 单帧缺失时序，动作决策抖动 |
| 通道换序 | `VecTransposeImage`（HWC->CHW） | 对齐 PyTorch 卷积输入布局 | 容易出现通道维报错 |
| 奖励归一化 | `VecNormalize(norm_reward=True)` | 缓和奖励尺度波动，提高 PPO 稳定性 | 更新步幅忽大忽小，曲线抖动 |
| 融合键校验 | 比较两模型 `state_dict` 键集合 | 保证参数可一一对齐 | 键不匹配导致错误融合 |
| 融合形状校验 | 同名参数必须形状一致 | 防止错误广播和隐式错配 | 结果不可控，性能随机波动 |
| 参数类型筛选 | 仅融合浮点参数 | 避免整型/buffer 被错误加权 | 统计状态异常，行为退化 |
| 融合范围控制 | `full` / `actor_feature` | 控制 critic 干扰，支持保守融合 | 价值头干扰 actor，迁移不稳 |
| alpha 解析 | `alpha in [0,1]` 且网格非空 | 保证融合系数合法可复现 | 参数非法，脚本直接失败 |
| 分数归一化 | `score_norm=minmax` 可选 | 防止高量级任务主导总分 | 最优 alpha 偏向数值大的任务 |
| mixed 空间校验 | 检查 obs shape/dtype/action space | 确保两环境可安全拼接 | 训练中途才暴露不一致错误 |
| runtime 值域校验 | reset 后检查 obs 值域与 NaN/Inf | 尽早发现预处理异常 | 进入训练后才出现梯度异常 |
| 评估日志分流 | `eval_mario/*`、`eval_coinrun/*` | 避免双任务曲线混写误判 | 图像“跳变”，难以解释结论 |

## 4. 分阶段详细说明

## 4.1 阶段 A：单任务输入预处理（融合前基础）

### 做法

- Mario：`SkipFrame -> GrayScale -> Resize(84x84)`，再在向量环境里 `VecFrameStack(4)`。
- CoinRun：通过 `ProcgenObsWrapper` 做灰度与缩放，并保持与 Mario 同步的跳帧策略。
- 训练前统一 `VecTransposeImage`，让 CNN 看到的是一致的通道布局。

### 原因

- 权重融合本质上默认了“两个专家的特征空间可对齐”。
- 如果两边输入分布在预处理层就不同（尺寸、布局、时序窗口不同），即便公式上能融合，也会得到不稳定的初始化策略。

### 设计原则

- 先统一输入语义，再讨论参数空间插值。
- 跨任务迁移时，优先保证“观测口径同构”。

---

## 4.2 阶段 B：融合前参数预处理（可融合性保证）

### 做法

- 读取两侧 `policy.state_dict()`。
- 严格校验：
  - 参数键集合一致；
  - 同名参数形状一致。
- 仅对浮点参数按公式融合：`W_fused = alpha * W_a + (1 - alpha) * W_b`。
- 非浮点参数直接继承 A（不做加权）。
- 支持 `fuse_scope`：
  - `full`：融合全部 policy 浮点参数；
  - `actor_feature`：仅融合特征提取器 + actor 分支。

### 原因

- 融合是逐元素线性插值，必须先保证元素能“正确对齐”。
- 将非浮点参数排除可避免 buffer 状态被错误插值。
- 在多任务冲突较强时，限制融合范围可降低 critic 相互干扰。

### 设计原则

- 先验证，再融合；失败要尽早失败（fail fast）。
- 融合不追求“参数都混”，而追求“对目标任务最稳”。

---

## 4.3 阶段 C：alpha 搜索与评分预处理（可比性保证）

### 做法

- 解析 `--alpha-grid`，要求都在 `[0,1]`。
- 每个 alpha 在 Mario/CoinRun 分别评估平均回报。
- 综合分支持两种：
  - `none`：直接加权原始回报；
  - `minmax`：先在 alpha 网格内对各任务回报做最小-最大归一化，再加权。

### 原因

- 双任务回报量级通常不同，直接相加会让大数值任务“天然占优”。
- 归一化能让“选择 alpha”更接近任务平衡目标，而不是量纲偏置。

### 设计原则

- 评分前先做任务可比性处理。
- 评价标准应与研究目标一致（保 Mario、稳 CoinRun，而不是单项极值）。

---

## 4.4 阶段 D：mixed 训练前一致性预处理（可训练性保证）

### 做法

`MixedVecEnv` 启动时会检查：

- `observation_space.shape` 一致；
- `observation_space.dtype` 一致；
- 动作空间一致；
- 首次 reset 后再检查 runtime obs 的维度、dtype、值域、NaN/Inf。

### 原因

- 静态空间一致不代表运行时一定一致（某些 wrapper 组合会在 runtime 暴露问题）。
- 先校验再训练能把错误暴露在第 0 步，而不是百万步后才崩。

### 设计原则

- 训练稳定性的第一步是“输入合法且同分布”。
- 混合环境必须具备显式防呆机制，而不是依赖人工假设。

---

## 4.5 阶段 E：mixed 训练中的奖励预处理（优化重心控制）

### 做法

- 默认路径（`use_aligned_reward=False`）：
  - Mario 保持原始奖励；
  - CoinRun 使用 `reward_scale + progress + success_bonus - fail_penalty - step_penalty`。
- 对齐路径（`use_aligned_reward=True`）：
  - CoinRun 使用统一模板 `progress + success - fail - step`；
  - 混合环境里 Mario 采用偏保守版本（前进项 + 通关项，不加失败/时间惩罚）。
总奖励 = 原始奖励缩放 + 右移增量奖励 + 通关奖励 - 失败惩罚 - 时间惩罚。
### 原因

- 多任务常见问题是奖励量级失衡，导致更新被单任务主导。
- 通过缩放和塑形，可把 CoinRun 梯度贡献调到与 Mario 更可控的区间。

### 设计原则

- 奖励预处理服务于优化平衡，不是简单“把分做高”。
- 训练目标是双任务可用，不是单任务极限分数。

---

## 4.6 阶段 F：评估日志预处理（可解释性保证）

### 做法

- 训练脚本中使用 `PrefixedEvalCallback`：
  - Mario 写入 `eval_mario/*`
  - CoinRun 写入 `eval_coinrun/*`

### 原因

- 两个 EvalCallback 写同一命名空间会造成曲线交替覆盖，看起来像“剧烈震荡”。
- 分流后才能准确判断是任务切换效应还是训练退化。

### 设计原则

- 指标口径必须与任务边界一致。
- 先保证可解释性，再谈调参优劣。

可写成一句话：

> 本研究在“单任务训练—参数融合—混合续训”全链路中采用分层预处理策略：首先统一观测口径（灰度、84x84、跳帧、帧堆叠与通道布局），其次在参数融合前执行键/形状/类型一致性校验并进行范围可控的线性插值，随后在 alpha 选择阶段引入任务级评分归一化，最后在 mixed 训练中实施输入空间一致性校验与奖励语义对齐。该策略提升了融合模型的可训练性、可复现性与结果可解释性。

## 6. 当前建议口径（保守版）

- 观测预处理：`84x84` 灰度 + `skip=4` + `frame_stack=4`。
- 融合范围：先从 `full` 起步；若出现迁移不稳，再试 `actor_feature`。
- alpha 评分：优先 `score_norm=minmax`，避免量级偏置。
- mixed 奖励：CoinRun 用可控塑形，逐步调小过强奖励项，防止压制 Mario。
- 评估日志：坚持 `eval_mario/*` 与 `eval_coinrun/*` 分流记录。

---

## 7. 风险提示（实验复现时务必关注）

1. 长时间训练成本高：`total_timesteps` 较大时，建议先小规模 smoke test（如 1e5）确认链路正确。
2. 口径漂移风险：任何一次改动 `skip/frame_stack/reward_scale/score_norm` 都应同步写入实验记录。
3. 评估污染风险：训练环境与评估环境必须保持分离，避免指标失真。
