# 权重融合到混合训练实验过程总结（论文版）

本文档总结本轮从“单任务 PPO 权重融合”到“Unified 混合训练二次微调”的全过程，重点记录每个阶段的配置变化、现象与结论，便于论文实验章节复现与撰写。

---

## 1. 实验目标

- 起点：已有 Mario 单任务 PPO 与 CoinRun 单任务 PPO。
- 目标：先做参数融合得到统一初始化模型，再在 mixed 环境中继续训练，提升 Mario，同时尽量保留 CoinRun 性能。

---

## 2. 阶段总览（时间线）

| 阶段 | 实验 ID / 产物 | 核心配置 | 关键现象 |
|---|---|---|---|
| A. 权重融合 | `best_model/fused/mario_coinrun_v1/best_model.zip` | 融合脚本 + 0.5/0.5 融合权重（终端记录） | 成功得到可用于 unified 续训的初始化模型 |
| B. 混合训练 v1 | `unified_from_fused_v1` | `mario_ratio=0.5`，`total_timesteps=10M`，CoinRun 默认缩放（200） | CoinRun 很快学满，Mario 奖励有提升但通关率长期 0 |

## 3. 关键过程与变化

## 3.1 阶段 A：先融合再训练（初始化策略）

- 采用权重融合流程，把 Mario 与 CoinRun 的单任务 PPO 参数融合为一个统一初始模型。
- 融合产物用于后续 `train_model/train_unified.py --resume ...`。
- 这一步的意义：降低从随机初始化直接做多任务训练的难度。

## 3.2 阶段 B：`unified_from_fused_v1`（首轮 mixed）

- 主要配置（终端与脚本默认推断）：
  - `mode=mixed`
  - `n_envs=10`
  - `mario_ratio=0.5`
  - `total_timesteps=10_000_000`
  - `fixed_level=True, start_level=0, distribution_mode=easy`
  - CoinRun 相关为脚本默认：`reward_scale=200, progress_coef=0.05, success_bonus=10, fail_penalty=2, step_penalty=0.002`
- 观测现象：
  - CoinRun 很快达到高完成率（接近 1）。
  - Mario 回报有提升，但长期 `mario_complete_rate=0`，说明“会拿分但不稳定通关”。

## 3.3 阶段 C：评估指标分流（可视化修正）

- 问题：Mario/CoinRun 两个 `EvalCallback` 都写入 `eval/mean_reward`，同图出现“高低跳变”。
- 处理：
  - 新增 `callbacks/prefixed_eval_callback.py`
  - 在 `train_unified.py` 中将两路评估分别写为 `eval_mario/*`、`eval_coinrun/*`
- 结果：TensorBoard 可正确按任务解读，不再把“多任务切换”误判为“训练崩溃”。

## 3.4 阶段 D：`unified_from_fused_v2`（提高 Mario 比例但仍不足）

- 配置变化：
  - `mario_ratio` 从 `0.5` 提到 `0.6`
  - `coinrun_reward_scale` 从默认 `200` 降到 `150`
- 观测现象（终端）：
  - CoinRun 仍接近学满。
  - Mario 指标改善有限，`mario_complete_rate` 仍未有效抬升。
- 解释：仅轻度调整采样比例和奖励权重，不足以改变更新重心。

## 3.5 阶段 E：修复续训超参数“被硬编码覆盖”

- 发现问题：`--resume` 分支会把 `learning_rate=3e-4`、`ent_coef=0.1` 硬覆盖，导致后期微调过于激进。
- 已实施改动（`train_model/train_unified.py`）：
  - 新增参数：
    - `--learning-rate`
    - `--learning-rate-schedule {constant, linear}`
    - `--ent-coef`
  - `resume` 与 `new train` 两条路径统一使用上述参数。
  - 对 `resume` 显式调用 `model._setup_lr_schedule()`，确保调度生效。

## 3.6 阶段 F：保守微调验证（当前）

- 当前策略：在“保留 CoinRun”的前提下，温和向 Mario 倾斜。
- 终端已观测到：
  - 学习率随训练进度下降（约 `1e-5` 量级）。
  - `eval_mario/mean_reward` 较此前阶段显著提高。
  - `eval_coinrun` 保持稳定（短回合高完成）。
- 说明：相比 v2，新的调度方式更符合“后期精调”目标。

---

## 4. 论文可写结论（可直接引用）

1. **先融合再 mixed 训练**可显著提升多任务训练起点，避免从随机初始化直接对齐两任务。  
2. **多评估源同名日志会造成错误判断**，必须分流 `eval_mario/*` 与 `eval_coinrun/*`。  
3. 在 CoinRun 已学会后，仅提升 `mario_ratio`（如 0.5→0.6）通常不够，需要同时控制优化强度。  
4. **续训阶段使用线性学习率衰减 + 适度降低熵系数**，比固定高学习率更适合“轻微提升 Mario 且保留 CoinRun”。  

---

## 5. 当前建议的复现实验设置（保守版）

- 目的：轻微提升 Mario，尽量不破坏 CoinRun。
- 建议参数：
  - `mario_ratio=0.8`
  - `coinrun_reward_scale=150`
  - `learning_rate=3e-4`
  - `learning_rate_schedule=linear`
  - `ent_coef=0.08`
- 记录项：
  - `eval_mario/mean_reward`, `eval_mario/mean_ep_length`
  - `eval_coinrun/mean_reward`, `eval_coinrun/mean_ep_length`
  - `rollout/mario_complete_rate`, `rollout/coinrun_complete_rate`

> 注：论文中请明确区分“回报提升”和“通关率提升”，两者在 Mario 任务上并不总是同步。
