# 单一训练预处理详解（Mario / Jumper / CoinRun）

> 目的：把单任务 PPO 训练中“观测与奖励是如何被处理的”讲清楚，方便复现实验、解释结果与排查问题。

---

## 1. 适用范围与入口脚本

本文对应以下单任务训练入口：

- `train_model/train_ppo_mario.py`
- `train_model/train_ppo_jumper.py`
- `train_model/train_ppo_coinrun.py`
- 以上入口最终都调用 `train_model/train_ppo_model.py`

也就是说，不同单任务的超参数可以不同，但主干预处理链路是一致思路：**先统一观测语义，再统一时序信息，再统一张量布局**。

---

## 2. 先看总流程（从环境到网络）

```text
原始游戏帧
  -> 跳帧（skip=4，动作重复+奖励累积）
  -> 灰度化（RGB -> 1通道）
  -> 缩放（84x84）
  -> 帧堆叠（4帧）
  -> 通道换序（HWC -> CHW）
  -> 可选奖励归一化（VecNormalize，默认仅reward）
  -> CustomCNN / CnnPolicy
```

---

## 3. 每一步预处理做了什么，为什么要做

## 3.1 跳帧（Skip Frame）

实现位置：

- Mario：`envs/skip_frame.py` + `envs/mario_env.py`
- Jumper/CoinRun：`envs/jumper_env.py` 中 `ProcgenObsWrapper._repeat_step()`

当前配置：

- `skip=4`

作用：

- 同一个动作连续执行 4 帧，只把窗口内累计奖励返回给策略。

原因：

- 降低决策频率，减少计算量，加快训练。
- 相邻原始帧通常差异较小，逐帧决策信息冗余较高。
- 与 Atari/PPO 常见设置一致，工程上更稳。

影响与注意点：

- 优点：训练更快、梯度噪声更小、收敛更稳定。
- 风险：动作控制粒度变粗，极端精细操作可能受影响。
- done 处理必须正确：窗口内某个子环境 done 后不应继续累计该子环境后续奖励（你当前 Procgen 包装器已考虑）。

---

## 3.2 灰度化（GrayScale）

实现位置：

- Mario：`GrayScaleObservation(env, keep_dim=True)`
- Procgen：`_grayscale_resize()` 中按加权系数转灰度

作用：

- 把 `RGB(3通道)` 转成 `灰度(1通道)`。

原因：

- 任务核心信息主要来自轮廓、位置、相对运动，不强依赖颜色。
- 降低输入维度，减少参数量与显存占用。
- 在样本效率有限时，减少无关特征有助于稳定学习。

影响与注意点：

- 优点：更省算力，常见场景下收敛更稳。
- 风险：若环境有“颜色即语义”的关键机制，灰度可能损失信息。
- `keep_dim=True` 很重要：保持通道维，便于后续堆叠与 CNN 输入一致。

---

## 3.3 尺寸缩放（Resize 到 84x84）

实现位置：

- Mario：`ResizeObservation(env, (84, 84))`
- Procgen：`_grayscale_resize(..., target_size=(84, 84))`

作用：

- 把不同来源分辨率统一到 `84x84`。

原因：

- 统一输入尺度，才能共享同一 CNN 结构与训练配置。
- `84x84` 是经典 RL 图像输入尺寸，信息密度和计算开销平衡较好。

影响与注意点：

- 尺寸太大：算力压力大、训练慢。
- 尺寸太小：细节丢失，障碍/边缘信息不清晰。
- 当前 `84x84` 对 Mario 与 Procgen 是工程上较稳的折中。

---

## 3.4 帧堆叠（Frame Stack）

实现位置：

- `envs/unified_env.py` 中 `VecFrameStack(..., n_stack=4, channels_order="last")`

作用：

- 把最近 4 帧灰度图沿通道维拼起来，形成 `84x84x4`。

原因：

- 单帧无法表达速度与方向（例如角色向右还是向左移动）。
- PPO 策略若只看静态图，容易出现“看得见但判断不出动态趋势”。

影响与注意点：

- 优点：显著提升动态判断能力。
- 代价：输入通道从 1 增加到 4，计算量上升。
- `skip=4 + stack=4` 组合本质上覆盖了一个短时历史窗口，是当前项目的关键时序信息来源。

---

## 3.5 通道换序（VecTransposeImage）

实现位置：

- `train_model/train_ppo_model.py`：`VecTransposeImage(vec_env)`

作用：

- 把环境输出从 `channels_last`（`HWC`）转换为 `channels_first`（`CHW`）。

原因：

- PyTorch 的卷积默认使用 `NCHW`，这样能减少额外转换与兼容问题。
- 保证和 `CnnPolicy` / `CustomCNN` 输入习惯一致。

影响与注意点：

- 不改变语义，只改变内存布局与计算接口。
- 若缺失这一步，常见报错是卷积通道数不匹配或输入形状不符合预期。

---

## 3.6 奖励归一化（可选 VecNormalize）

实现位置：

- `train_model/train_ppo_model.py`，通过 `--normalize-reward` 开关控制

当前逻辑：

- `norm_obs=False`
- `norm_reward=True`
- `clip_reward` 默认 10.0

作用：

- 对奖励做归一化和裁剪，减少回报尺度波动。

原因：

- PPO 对奖励尺度较敏感；归一化可提升训练稳定性，尤其是长时训练。
- 只归一化 reward、不过度改动 obs，可避免与图像预处理重复叠加带来的不确定性。

影响与注意点：

- 优点：训练曲线更平滑，更新更稳定。
- 风险：训练和评估若混用统计，可能导致指标解释偏差。
- 你的实现已经把 train/eval 环境分离，方向是正确的。

---

## 4. Mario / Jumper / CoinRun 三者差异对比

| 项目 | Mario | Jumper (Procgen) | CoinRun (Procgen) |
|---|---|---|---|
| 跳帧 | `SkipFrameWrapper` | `ProcgenObsWrapper._repeat_step` | `ProcgenObsWrapper._repeat_step` |
| 灰度+缩放 | Gym Wrapper | 自定义函数 `_grayscale_resize` | 自定义函数 `_grayscale_resize` |
| 帧堆叠 | `VecFrameStack=4` | `VecFrameStack=4` | `VecFrameStack=4` |
| 通道换序 | `VecTransposeImage` | `VecTransposeImage` | `VecTransposeImage` |
| 奖励归一化 | 可选 | 可选 | 入口默认更常开启 |
| 额外奖励处理 | 无统一奖励模板 | 无统一奖励模板 | 可选 `use_aligned_reward` |

结论：

- 观测预处理链路已经高度统一；
- CoinRun 在奖励语义上有额外对齐策略，这是单任务差异的主要来源之一。

---

## 5. 为什么这些预处理对“单一训练”重要

## 5.1 可复现性

- 输入尺寸、通道顺序、时间窗口固定后，不同实验间更可比。
- 论文或答辩中可明确说明“模型输入口径完全一致”。

## 5.2 稳定性

- 跳帧与奖励归一化降低训练噪声；
- 帧堆叠补足动态信息，减少策略震荡。

## 5.3 工程可扩展性

- Mario 与 Procgen 统一到同样观测语义，后续做 mixed/unified 更自然。
- 便于共享 `CustomCNN`、共享 checkpoint 与评估脚本。

---

## 6. 常见错误与排查清单

1. 形状错误（最常见）
   - 现象：卷积层报通道不匹配。
   - 排查：确认 `VecFrameStack` 在前、`VecTransposeImage` 在后，并检查输入是否 `NCHW`。

2. 训练快但策略发散
   - 排查：看是否误关 `frame_stack` 或改动了 `skip` 导致时序信息不足。

3. 不同实验难以比较
   - 排查：确认是否混用了不同分辨率/不同奖励口径/不同归一化设置。

4. 评估结果异常波动
   - 排查：确认 train/eval 是否严格分离，评估环境不要继承训练期统计污染。

---

## 7. 当前项目建议口径（用于文档与答辩）

- 观测口径：`84x84` 灰度，`4` 帧堆叠；
- 时序口径：`skip=4` 动作重复与奖励累积；
- 张量口径：训练前统一转为 `CHW`；
- 奖励口径：默认原始奖励，CoinRun 可选统一奖励模板与奖励归一化；
- 评估口径：train/eval 分离，避免统计与状态污染。
